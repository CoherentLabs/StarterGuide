<Summary>
Gameface's layout engine runs entirely on <Highlight>Yoga</Highlight>, a flexbox-only algorithm. There is no CSS Grid. The engine classifies every CSS change as either a <Highlight>Full Layout Solve</Highlight> or a lightweight <Highlight>Transform Update</Highlight> on each frame — and that classification determines whether your UI runs at a steady 60 FPS or stutters. On top of that, the layout algorithm has <Highlight>exponential complexity</Highlight> relative to DOM depth under specific conditions. This article explains what triggers the expensive path, what the performance cliff looks like in practice, and the two CSS changes that eliminate it.
</Summary>

## The Flexbox Foundation

Gameface uses the <Highlight>Yoga layout engine</Highlight> (developed by Meta/Facebook) as its sole layout algorithm. Yoga implements the CSS flexbox standard. No other layout model is available.

`display: grid` is entirely absent. All grid-related CSS properties (`grid-template-columns`, `grid-area`, `grid-gap`, and every other `grid-*` variant) are silently ignored by the renderer. Any layout built on CSS Grid must be rebuilt with flexbox equivalents.

Display values like `inline-block`, `inline-flex`, `contents`, and `inline-grid` are also unsupported and will be rejected. The only supported display values for layout work are `flex`, `none`, and `inline`.

HTML elements that a browser renders as block-level containers are internally mapped to flex containers by the engine, using a set of equivalent flex properties. Every element in Gameface operates under flex rules — either as a declared flex container or as a simulated one.

This uniform model works in your favor for game UI layouts. Absolute positioning and flexbox compose cleanly, making it straightforward to layer HUDs, tooltips, and overlays. The pattern below is a reliable starting point for any positioned UI panel:

```html
<div class="hud">
    <div class="hud__health-bar"></div>
    <div class="hud__ammo-counter"></div>
</div>
```

```css title="hud.css"
.hud {
    display: flex;
    position: absolute;
    bottom: 2rem;
    left: 2rem;
    gap: 1rem;
    align-items: flex-start; /* Explicit non-stretch value — see performance section */
}

.hud__health-bar {
    flex-basis: 20rem;       /* Defined size — critical for performance */
    height: 1.5rem;
}
```

## Transforms vs. Full Layout Solves

The engine analyzes what changed in the DOM after each JavaScript and animation tick and classifies the required layout work into one of two paths.

A <Highlight>Full Layout Solve</Highlight> runs whenever a property that affects the document flow has changed. This includes `width`, `height`, `margin`, `padding`, `flex-basis`, `flex-grow`, `flex-shrink`, `left`, `right`, `top`, and `bottom`. The engine invokes Yoga, which traverses the affected subtree and recalculates positions and sizes from scratch.

A <Highlight>Transform Update</Highlight> runs when only `transform` or `opacity` has changed. The untransformed positions of elements have not shifted, so Yoga is skipped entirely. The engine runs a lighter pass that recalculates bounding boxes and visual positions without touching the layout tree.

The CPU cost difference is significant. A panel animated by its `left` property forces a full Yoga traversal on every frame. Changing it to animate `transform: translateX` instead skips Yoga completely.

Floating damage numbers are a common game UI pattern and a clear demonstration of this trade-off. The bad version animates `top`, triggering a full layout solve per frame. The good version animates `transform: translateY` and stays in the lightweight path:

```html
<div class="damage-number">-245</div>
```

```css title="hud.css"
/* ❌ Animating 'top' — triggers a full layout solve on every frame */
@keyframes float-bad {
    from { top: 0;      opacity: 1; }
    to   { top: -5rem;  opacity: 0; }
}

/* ✅ Animating 'transform' — lightweight transform update only, Yoga is skipped */
@keyframes float-good {
    from { transform: translateY(0);     opacity: 1; }
    to   { transform: translateY(-5rem); opacity: 0; }
}

.damage-number {
    position: absolute;
    animation: float-good 0.8s ease-out forwards;
}
```

## The Layout Performance Cliff

Yoga's algorithm distributes space across two axes and must traverse the layout tree multiple times per node to do so. The number of traversals depends on how much information is already known about each node's size. Under ideal conditions, each node requires one pass. Under the worst conditions, each node triggers up to four recursive calls into its children.

The worst-case complexity is <Highlight>O(4^Depth)</Highlight>. A tree with 8 nodes and a depth of 4 produces 172 layout calls per frame in the worst case. The same tree produces 9 calls in the best case. Real game UIs measured across multiple shipped Gameface titles show DOM depths between 14 and 28 nodes, with total node counts ranging from 125 to nearly 7,000. An unoptimized subtree at that depth generates thousands of redundant calls per frame.

The worst case requires all three conditions to be true simultaneously:

**1. No defined flex dimensions.** Children without a defined `flex-basis`, `width`, or `height` force Yoga to descend into their subtree just to measure their natural size (the internal `measure` call).

**2. Cross-axis stretching.** `align-items: stretch` forces Yoga to re-enter each child's subtree after measuring, to grow it to fill the cross-axis. This generates the `stretch` and `multiline-stretch` recursive calls.

**3. Wrapping enabled.** `flex-wrap: wrap` causes the algorithm to loop through lines of items, re-evaluating placement on each pass.

The following CSS represents the dangerous pattern, using an inventory grid as the concrete example:

```html
<!-- An inventory grid with many item slots -->
<div class="inventory-grid">
    <div class="item-slot">
        <div class="item-slot__icon"></div>
        <div class="item-slot__label">Iron Sword</div>
    </div>
    <!-- dozens more slots... -->
</div>
```

```css title="inventory.css"
/* ❌ All three worst-case conditions are active */
.inventory-grid {
    display: flex;
    flex-wrap: wrap;          /* Condition 3: wrapping enabled */
    /* align-items defaults to 'stretch' — condition 2 is active without writing it */
}

.item-slot {
    display: flex;
    flex-direction: column;
    /* No flex-basis, no width — condition 1: Yoga must enter each slot to measure it */
}
```

:::danger[align-items: stretch Is the CSS Default]

`align-items: stretch` is the CSS specification default. You do not need to write it — any flex container without an explicit `align-items` declaration is already using it. A deep, wrapping UI tree that looks like clean code can be silently running in worst-case performance mode on every frame. If you are not sure whether a container needs stretching, declare `align-items: flex-start` explicitly.

:::

## How to Optimize Layouts

Removing the worst-case conditions requires eliminating the recursive calls that Yoga makes into child subtrees. Both fixes below can be applied independently, and their effects are cumulative.

### Define Dimensions on Flex Items

A defined `flex-basis` (or explicit `width`/`height`) on every flex item eliminates the `measure` recursive call. Yoga no longer needs to enter the child's subtree to figure out its size — the answer is already in the node.

The fix applies at every level of the tree, not only on leaf nodes. Defining sizes on wrapper containers has the same benefit: their subtrees are visited once instead of repeatedly:

```css title="inventory.css"
/* ✅ Defined dimensions eliminate the 'measure' call at every level */
.inventory-grid {
    display: flex;
    flex-wrap: wrap;
    align-items: flex-start;
}

.item-slot {
    display: flex;
    flex-direction: column;
    align-items: flex-start;
    flex-basis: 8rem;          /* Defined — Yoga skips the measure pass for this subtree */
    height: 10rem;
}

.item-slot__icon {
    width: 6rem;               /* Leaf node size defined — no recursive measure needed */
    height: 6rem;
}

.item-slot__label {
    flex-basis: 8rem;          /* Width matches the slot — Yoga can skip measuring text wrapping */
}
```

The performance impact is not marginal. Gameface's own internal stress tests, run on a realistic UI structure similar to shipped client projects (4,400+ nodes, depth of 21), show a layout time of ~2,000ms on Windows when only leaf nodes have defined sizes. The same structure with flex-basis defined on both wrappers and leaves runs at ~17ms. That is a reduction of over 99% for the same visual output.

### Use align-items Sparingly

Each `align-items: stretch` on a flex container adds two extra layout calls per line per frame (`stretch` and `multiline-stretch`). The more children and the more wrapping, the more these accumulate across the tree.

The fix is to replace `stretch` with an explicit `flex-start`, `flex-end`, or `center` on containers where children do not need to fill the cross-axis. Reserve `align-items: stretch` for the specific containers that genuinely require it:

```css title="inventory.css"
/* ❌ 'stretch' is the default — three containers silently trigger cross-axis traversals */
.inventory-grid  { display: flex; flex-wrap: wrap; }
.item-slot       { display: flex; flex-direction: column; }
.item-slot__info { display: flex; }

/* ✅ Explicit non-stretch values — cross-axis recursive calls eliminated */
.inventory-grid  { display: flex; flex-wrap: wrap;        align-items: flex-start; }
.item-slot       { display: flex; flex-direction: column; align-items: flex-start; }
.item-slot__info { display: flex;                         align-items: center; }
```

:::note[The Combined Effect: From 172 Calls to 9]

Both fixes together (defined dimensions on all flex items and explicit non-stretch `align-items`) reduce Yoga's traversal from O(4^Depth) to O(N) — a single pass. For the 8-node, depth-4 reference tree, the call count drops from 172 to 9. For a real UI at depth 19 with 2,500 nodes, this difference determines whether the layout phase consumes milliseconds or hundreds of milliseconds per frame.

:::
