## How Layout Works in Gameface

If you are coming from traditional web development, the biggest mental shift when authoring for Gameface is understanding the layout engine. 

In a standard browser, you have multiple layout algorithms: `block`, `inline`, `grid`, `flex`, and `table`. In Gameface, the layout engine relies exclusively on the <Highlight>Flexbox model</Highlight>. 
Even if you use alternative display types like `display: block`, Gameface simulates them under the hood using `flex` properties. 

Because everything is fundamentally flex-based, every time an element's size, position, or content changes, the engine evaluates how that change affects the surrounding document. 
This calculation process is called a <Highlight>layout pass</Highlight>. Because flexbox distributes space dynamically, elements are highly dependent on their siblings and parents. If one box changes its width, the engine calculates if the boxes next to it need to shrink, stretch, or wrap to a new line.

Gameface's internal layout engine is highly optimized and incredibly fast. However, the environment it runs in is extremely demanding. If a game targets 60 frames per second, the entire game engine has exactly <Highlight>16.6 milliseconds</Highlight> to render a single frame. The UI is usually allocated only a tiny fraction of that time (often 1-2ms). 

In a standard web browser, a 15ms layout recalculation goes completely unnoticed. In a game engine, it blows your entire frame budget. To ensure your UI runs at peak performance and leaves plenty of CPU room for the game itself, we need to architect our layouts following a few core rules.

## High-Performance Layout Rules

Before we look at the screen structure, here are the practical "Do's and Don'ts" for styling your elements in Gameface.

### Define explicit dimensions
When an element's size is left `undefined` (or set to `auto`), the engine does not know how big it is. 
It has to pause, dive into the element's children, measure all of their text and content, bubble that size back up, and *then* calculate how to fit it into the flex container. 

:::tip[The explicit size rule]
Always set a `flex-basis`, `width`, or `height` on your elements whenever possible. 
By explicitly defining the boundaries, you tell the engine exactly how much space to allocate immediately, entirely skipping the heavy measuring phase.
:::

### AVOID `align-items: stretch`
In standard web development, stretching items to fill a container is very common. In Gameface, `align-items: stretch` forces the engine to run multiple layout passes. 
It has to measure the children to find the largest one, and then run another pass to force the remaining children to grow and match that size.

Try to use `align-items: center` or `flex-start` and rely on explicit dimensions instead.

:::danger[Default property]
Remember that `align-items` defaults to `stretch`. If you forget to set it, you might accidentally trigger this expensive behavior without realizing it.
:::

### Absolute Positioning for Anchors
Use `position: absolute` for your top-level widgets. 
Absolute positioning removes the element from the normal flow and is computed separately. If an absolutely positioned widget changes its internal size, it will never push its siblings around.

**Let's look at practical examples of when to use this:**

#### Scenario 1: Centering a Crosshair
If you place a static crosshair in the middle of the screen using flexbox, it won't hurt performance much because it never changes. However, in most games, a crosshair is dynamic—it scales up (reticle bloom) 
when the player shoots or moves.

If the crosshair is in a flex container, animating its size might trigger a screen-wide layout update. 
To solve this, we remove the crosshair from the document flow entirely using `position: absolute`. 
We can then use CSS transforms to pinpoint its location directly in the center of the screen without affecting any other UI elements. 

```html title="hud.html"
<body>
    <div class="crosshair-wrapper">
        <img class="crosshair" src="crosshair.png" alt="Crosshair" />
    </div>
</body>
```

```css title="hud.css"
.crosshair-wrapper {
    /* Removed from the flex flow */
    position: absolute;

    /* Always define explicit dimensions */
    width: 2rem;
    height: 2rem;
    
    /* Move the top-left corner of the element to the center of the screen */
    left: 50%;
    top: 50%;
    /* Pull the element back by half its own width and height to center it perfectly */
    transform: translate(-50%, -50%); 
}
.crosshair {
    /* Animating scale for bloom effect */
    transition: transform 0.1s ease-out;
    width: 100%;
    height: 100%;
    transform: scale(1.1);
}
```

:::note[Why It Works?]
When the crosshair scales up or down using transforms, the layout engine ignores it. It simply repaints the larger crosshair independently, preserving your frame budget.
:::

#### Scenario 2: Anchoring to a 3D Object
When creating a nameplate that floats above a 3D character's head, the game will send updated 2D screen coordinates (left/top) to the UI every single frame as the camera moves.

Passing these constant updates to an element inside a flex container would force the engine to recalculate the layout repeatedly. 
Instead, we use `absolute positioning` so the engine can independently paint the nameplate at the new coordinates while bypassing the rest of the layout.

```html title="hud.html"
<body>
    <div class="nameplate">
        Player 1
    </div>
</body>
```

```css title="hud.css"
.nameplate {
    /* Absolute is mandatory here */
    position: absolute; 
    
    /* Explicitly size the container */
    width: 15rem;
    height: 3rem;

    /* The Left/Top values are injected by JavaScript every frame */
    left: 0;
    top: 0;
}
```

:::note[Why It Works?]
Updating `left` and `top` coordinates on an absolutely positioned element only updates that specific node.
 If this nameplate were part of a flex grid, updating its position 60 times a second would force continuous, heavy layout recalculations.
:::

### Flexbox for Widget Alignment
While absolute positioning is perfect for root widgets, it is highly impractical for laying out dynamic content *inside* a widget (like lists, rows, or grids). This is where you should use `display: flex`.

#### Scenario 3: Centering within a control
Let's align an icon and text perfectly inside an action button. Because these internal elements dictate the visual flow of the button itself, `display: flex` is the perfect tool. 
To keep it performant, we will explicitly define the boundaries of the button to prevent the engine from recursively measuring it.

```html title="button.html"
<div class="action-button">
    <div class="action-icon"></div>
    <span class="action-text">Attack</span>
</div>
```

```css title="button.css"
.action-button {
    display: flex;
    /* Define explicit boundaries to avoid recursive measuring */
    width: 15rem; 
    height: 4rem;
    
    /* Center securely without using 'stretch' */
    justify-content: center;
    align-items: center; 
    gap: 1rem;
}

.action-icon {
    /* Explicit sizing for flex children */
    width: 2rem;
    height: 2rem;
    background-color: white;
}
```

#### Scenario 4: Anchoring to a UI element
Sometimes you need to mix both approaches. For example, you might want to attach a red notification badge (like a "New Item" dot) to the corner of an inventory slot. 

By setting the parent slot to `position: relative`, we create a local coordinate system. 
This allows us to safely use `position: absolute` on the badge, pulling it out of the flex flow while keeping it visually anchored to the item.

```html title="inventory.html"
<div class="inventory-slot">
    <div class="item-icon"></div>
    <div class="notification-badge">3</div>
</div>
```

```css title="inventory.css"
.inventory-slot {
    /* display: flex controls the internal item image */
    display: flex;
    width: 5rem;
    height: 5rem;

    /* position: relative creates a local anchor point for absolute children */
    position: relative;
}

.notification-badge {
    /* Absolute removes it from the inventory slot's flex flow */
    position: absolute;
    width: 1.5rem;
    height: 1.5rem;

    /* Push it slightly outside the top right corner */
    top: -0.5rem;
    right: -0.5rem;
}
```

:::note[Why It Works?]
Notification badges toggle on and off frequently. Because the badge is absolutely positioned, hiding or showing it does not force the surrounding inventory grid to recalculate its flex spacing.
:::

:::tip[Advanced Optimization: Simple Layout]
If your UI consists *entirely* of absolutely positioned elements (like a pure HUD), Gameface offers a specialized mode called `display: simple` that you can apply to the `<body>`. It acts as an ultra-fast rendering mode designed specifically for absolutely positioned widgets. We will explore this fully in the advanced **Phase 5: Polishing** section of the guide!
:::
