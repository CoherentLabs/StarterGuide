## RectTransforms vs. the CSS Box Model

In Unity, every UI element has a **RectTransform** that describes its position, size, and anchor relative to its parent canvas. In Gameface, layout is handled by the standard **CSS Box Model** combined with CSS Flexbox.

**Key mental shift:** You no longer think in terms of anchors and pivot points. Instead, you compose layouts by nesting flex containers.

| Unity concept | Gameface equivalent |
|---|---|
| RectTransform anchors | `position: absolute` + CSS `top/left/right/bottom`, or a flex parent |
| Canvas Scaler | Root `font-size` rem scaling (see [Building Scalable & Responsive Game UIs](/layout-assets-and-styling/styling--scalability/building-responsive-game-ui/)) |
| Layout Group (Horizontal/Vertical) | `display: flex` with `flex-direction: row` or `column` |
| Grid Layout Group | CSS Grid - note Gameface uses Yoga (Flexbox only); CSS Grid is not supported |

## Prefabs vs. Web Components / Frameworks

Unity Prefabs let you create reusable, self-contained UI widgets. In Gameface, reusability is achieved through:

- **[Gameface UI component library](https://gameface-ui.coherent-labs.com/)** - pre-built, production-ready UI components you can drop into any project.
- **SolidJS / React components** - the framework component model, where a `.tsx` file is the equivalent of a Prefab.
- **Web Components** - standard browser custom elements work in Gameface.

## MonoBehaviours vs. Frontend State + Engine Events

In Unity, C# MonoBehaviours attached to a Canvas handle UI logic. In Gameface:

- **UI logic** (opening menus, playing animations, toggling states) lives in **JavaScript/TypeScript** on the frontend.
- **Game logic** (damage calculation, inventory management) stays in **C++/C# on the engine side**.
- Communication between the two uses [Engine Communication: Setup & Events](/logic-and-interactions/engine-communication--data-mocking/engine-communication/).
